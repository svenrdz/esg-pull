    esgpull is a tool that simplifies usage of the ESGF Search API for data discovery, and manages procedures related to downloading and storing files from ESGF.

    Glossary
    Dataset
    collection of files. Described by metadata that follows its project's conventions.
    Facet
    basic element of a dataset's metadata. Pair of strings in the form name:value, equivalent to a python dictionary's item.

    Initialisation¶

    Before anything, make sure esgpull is correctly installed. Then you can run this next command to fill the database with all facets that can be found in ESGF index nodes.

    $ esgpull facet init

    Data discovery¶

    There are two ways to search for one or more datasets: exact and free-text search. Both are done through the esgpull search command.

    Exact search¶

    Exact search is done by providing a facet, such as <name>:<value>, for which both are matched exactly.

    Free-text Search¶

    Free-text search is done on anything you write that is not using the name:value syntax. Each word will be sent to the ESGF search API, which uses Apache Solr to match results.

    Solr syntax

    It's important to know that Solr will try to match any of the search terms.

    esgpull search surface temperature # will match either `surface` or `temperature`

    To match all the words, they must be written between quotes and separated with the keyword AND: 

    esgpull search "surface AND temperature" # will match BOTH `surface` and `temperature`

    Wildcard syntax¶

    A wildcard * can be used in both exact and free-text search, allowing for more controllable search results.

    Note that on most shells, the wildcard symbol should be inside " quotes, to escape it from being expanded by the shell before esgpull receives any input.

    All initializations for areacella variable from piControl experiments
    esgpull search "member_id:r1i*p1f1" table_id:fx variable_id:areacella experiment_id:piControl

    Case sensitivity¶

    For exact search, the case must match exactly, so project:cmip6 is going to have very different results than project:CMIP6.

    In the case of free-text search, the opposite is true, case is entirely insensitive, and may lead to funky results.

    Selection files¶

    When a search command becomes unusable from using a lot of long terms, selection files should be used. They are more flexible, and can combine multiple requests into one, using subqueries.

    Subqueries are extensions of the main query. Each subquery inherits all terms from the latter, and replaces any exact term using the same facet name. It is possible to append homonym terms, instead of replacing them, by prefixing the name with a + symbol.

    A selection file is written using either Yaml/1.0 or Json, Yaml is recommended for its human-readable syntax.

    Syntax Rules

    1. key/value pair ⇒ exact search term
      <name>: <values>
    2. query keyword ⇒ free-text
      query: <free-text>
    3. requests keyword ⇒ subqueries
      requests:
  - <subquery_1>
  - <subquery_2>
    4. a subquery contains only exact or free-text terms
      requests:
  - <name_1>: <values_1>
    <name_2>: <values_2>
    query: <free-text>
    5. a subquery inherits all of the main query's terms
      Main with one subquery
      <name_1>: <values_1>
requests:
  - <name_2>: <values_2>
      Equivalent flat syntax
      <name_1>: <values_1>
<name_2>: <values_2>
    6. an exact search term name prefixed with + is appended to the main query's homonym term
      Main with two subqueries
      <name_1>: <values_1>
requests:
  - <name_2>: <values_2>
  - +<name_1>: <alt_1>
      Equivalent expanded without
      requests:
  - <name_1>: <values_1>
    <name_2>: <values_2>
  - <name_1>: <values_1>,<alt_1>

    Subquery expansion¶

    Before expansion
    <name_1>: <value>
requests:
  - <name_1>: <value_1>       # replaces <name_1> from the main query
    <name_2>: <value_2>
    query: <free-text>

  - +<name_1>: <alt_value_1>  # <alt_value_1> is appended to <name_1> from the main query
    <name_3>: <value_3>       #  > another subquery
    <name_4>: <value_4>       # /
    After expansion

    Selection file for IPCC experiments on surface temperatures

    surface_temperature.yaml
    query: "surface AND temperature"
requests:
  - project: CMIP5
    experiment: "rcp*"
  - project: CMIP6
    experiment_id: "ssp*"
    Search with selection file
    esgpull search --selection-file surface_temperature.yaml --one
